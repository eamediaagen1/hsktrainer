  - risk level: critical / high / medium / low

Audit scope:

A. Authentication and accounts
Check:
- secure password hashing using a modern password hashing algorithm, not plain hashes or reversible encryption
- secure session handling
- secure cookie flags: HttpOnly, Secure, SameSite
- no session fixation
- logout invalidates sessions correctly
- email verification flow
- password reset tokens are strong, single-use, and expire quickly
- brute-force protection on login, signup, reset, and magic-link endpoints
- rate limiting and abuse prevention
- no user enumeration through login, signup, or reset error messages
- MFA readiness for admin accounts
- admin accounts separated from normal user accounts where possible

B. Authorization
Check:
- users can only access their own progress, quiz results, billing state, profile, and study history
- no IDOR/BOLA issues on API routes, database queries, URLs, or client-side requests
- premium content is enforced server-side, not just hidden in frontend code
- free users cannot unlock HSK2+ via modified requests, local storage, hidden params, or direct API calls
- admin endpoints are protected and not discoverable or exposed accidentally
- role checks exist on every sensitive backend route

C. Input handling and common web vulns
Check for:
- XSS, especially anywhere user-generated content, profile data, notes, quiz answers, or query params are rendered
- injection risks: SQL, NoSQL, ORM injection, command injection
- CSRF on state-changing actions if cookie-based auth is used
- unsafe file upload handling if uploads exist
- open redirects
- SSRF if any URL fetch/import feature exists
- template injection
- insecure deserialization
- path traversal
- prototype pollution if relevant to the stack
- unsafe use of eval, innerHTML, dangerouslySetInnerHTML, or equivalent

D. Frontend/browser security
Check:
- strong Content Security Policy
- security headers including at least HSTS, X-Content-Type-Options, and frame protections where appropriate
- no secrets exposed in frontend bundles
- no privileged logic enforced only in JavaScript
- localStorage/sessionStorage not used for sensitive long-lived secrets unless justified
- third-party scripts minimized and trusted
- analytics/chat widgets do not expose sensitive data

E. Payments and subscriptions
Assume Stripe unless I tell you otherwise.
Check:
- I do not handle raw card data directly if using Stripe-hosted flows
- webhook signatures are verified correctly
- webhook events are idempotent
- subscription state is updated from trusted server-side events, not client redirects alone
- premium access is revoked correctly on failed/canceled subscriptions
- no price tampering from the client
- product/price IDs are validated server-side
- test and live keys are separated
- secrets are not committed to git or exposed in logs
- refunds/cancellations/customer portal flows are safe

F. API and backend security
Check:
- all APIs require proper authentication and authorization
- rate limiting exists on sensitive endpoints
- request validation exists on every input
- verbose error messages do not leak stack traces, SQL errors, secrets, or internal paths
- logs do not store passwords, tokens, card info, reset links, or sensitive PII
- CORS is locked down
- server-side secrets come only from environment/secret manager
- background jobs and cron tasks are authenticated and protected
- webhook endpoints are isolated and validated

G. Data protection and privacy
Check:
- minimal data collection
- sensitive data encrypted or otherwise protected at rest where appropriate
- backups are protected
- retention/deletion policy exists
- users can delete accounts or request deletion if required
- email addresses and learning progress are not publicly exposed
- privacy policy / terms / refund policy gaps that create operational risk

H. Infrastructure and deployment
Check:
- HTTPS only
- production environment variables configured safely
- least-privilege database credentials
- production database not publicly exposed
- staging/dev services not publicly accessible or reusing prod secrets
- secure CI/CD
- dependency and package audit
- lockfile present
- outdated/vulnerable packages flagged
- container hardening if Docker is used
- cloud storage bucket permissions checked
- domain, DNS, and email security basics checked
- monitoring/alerting/logging configured for auth spikes, 500s, payment failures, and suspicious activity

I. Secrets management
Check:
- no API keys, tokens, private keys, or webhook secrets in repo history
- secret rotation plan exists
- separate secrets for dev/staging/prod
- principle of least privilege for all API keys
- no secrets in frontend env variables unless explicitly public-safe

J. Abuse and business logic
Check:
- signup abuse and bot creation
- coupon abuse if coupons exist
- trial abuse using multiple accounts
- scraping of premium content
- quiz or progress endpoints being spammed
- email abuse from invite/reset/contact forms
- account sharing scenarios
- mass-download of vocabulary content
- bypass of premium gating through race conditions or stale cache

K. Observability and incident readiness
Check:
- audit logs for admin and billing actions
- alerts for repeated failed logins, unusual signup spikes, and webhook failures
- basic incident response plan
- rollback plan
- backup restore sanity check
- health checks for critical services
